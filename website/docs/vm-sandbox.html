VM Sandbox

Run terminal sessions inside a lightweight Linux VM so agents can't touch what they shouldn't.

Why sandbox?

Git worktrees isolate your code, but a terminal session still has full access to your host — your home directory, credentials, system binaries, network. When CLI agents run arbitrary commands, that's a lot of trust.

The VM sandbox wraps each terminal in a Lima virtual machine. Your project worktree is mounted into the VM (read-write), but the rest of your filesystem is walled off. Inside the worktree, every path your .gitignore matches — .env files, secrets/, node_modules, build artifacts — is masked with an empty placeholder so the agent only sees what git would track. The agent can install packages, run builds, and execute code without risk to your host or your host-only secrets.

How it works

  1. Enable sandbox on a task — Sandboxing is per-task, not per-project. When you start or continue a task, the run dialog has a Sandbox toggle. Flip it on and the task's terminal will spawn inside the VM.
  2. First terminal — Ouijit lazily creates an Ubuntu 24.04 VM via Lima on the first sandboxed spawn for a project. One VM per project, shared across that project's sandboxed tasks. Subsequent terminals reuse the running VM.
  3. Spawn progress — The spawn dialog shows each step: checking VM status, creating / starting, SSH connect, provisioning, isolating gitignored paths, launching shell. First boot takes a minute or two; later spawns are fast.
  4. Shell sessions — Terminals run inside the VM via limactl shell. Your worktree is mounted at the same path as on the host, so relative paths and git operations work normally.
  5. VM lifecycle & resources — Manage the VM from Project Settings → Sandbox. You can see the VM status (Running / Stopped / Broken / Not created), create, start, stop, or recreate it, open a VM console, and edit the merged Lima YAML config (memory, CPU, disk, mounts). Changing resources prompts you to recreate the VM.

Gitignored paths are masked

When a sandboxed terminal starts, every path your .gitignore matches is replaced with an empty placeholder inside the VM. Host-only secrets like .env or config/secrets.yml aren't visible to the agent, and platform-specific artifacts like macOS node_modules don't leak into the Linux guest.

There's no config file or allowlist to maintain. Ouijit runs git ls-files against the worktree and honors whatever git says is ignored — negations (!keep.env), nested .gitignore files, globs, and core.excludesFile all work. The terminal startup banner reports what was hidden, e.g. “Isolating 12 paths (7 dir, 5 file)”.

What this buys you

  • Secret isolation — gitignored .env, .npmrc, secrets/, and similar files stay on the host.
  • No ABI mismatches — darwin-arm64 node_modules, Rust target/, and Python .venv never collide with the Linux guest's toolchain.
  • Per-task isolation — two parallel sandboxed tasks on the same project can't stomp each other's installs.

Caveats

  • Committed files are always visible. .gitignore only affects files git considers untracked. If you committed a secret and then gitignored it, the agent still sees it — run git rm --cached <file> and rotate the secret. Don't commit what you don't want an agent to read.
  • Masking is a snapshot at terminal start. New gitignored files created while a terminal is already open aren't masked in that terminal. Close and reopen to re-enumerate.
  • Filenames aren't hidden, only contents. ls still shows .env.secrets as a 0-byte file. Don't rely on this to hide secret-sounding paths.
  • The worktree is read-write. The VM has no git credentials, so a compromised agent can't push — but it can stage commits that a human on the host could unknowingly push later. Review diffs before pushing.
  • Scope is the worktree. Files in the VM's own $HOME (e.g. ~/.netrc, ~/.git-credentials) aren't covered. Lima ships a clean home; don't copy host auth into it.

Startup banners

Masking is fail-closed: if isolation can't be delivered, the terminal refuses to start. Every spawn resolves to one of three banners, printed to the terminal before the interactive shell:

  • SANDBOX ISOLATION ACTIVE — green. Masks applied. Shell starts.
  • SANDBOX NO PATHS TO ISOLATE — yellow. Nothing was gitignored in the worktree. Shell still starts — double-check your .gitignore before running anything sensitive.
  • SANDBOX ISOLATION FAILED — red. Enumeration error, missing sudo, mount failure, or any partial failure. The shell exits immediately; fix the cause and reopen. Partial failures are treated as total failures.

Errors that happen before the VM side runs (e.g. git ls-files throws) come back as a failed terminal-open in Ouijit, not as a banner.

Customizing the VM

VMs start from a clean Ubuntu 24.04 image with bash, git, curl, wget, nodejs, npm, python3, and build-essential pre-installed. Defaults are 4 GiB memory, 2 CPUs, 10 GiB disk.

Everything else — resource limits, extra packages, additional mounts, provisioning scripts — is controlled by the Lima YAML config in Project Settings → Sandbox. The editor shows the merged effective config and lets you write overrides; Ouijit prompts you to recreate the VM when changes require it.

See the Lima configuration reference for the available keys.

Claude Code integration

Ouijit automatically injects its hook helper script and Claude Code settings into each sandboxed shell. This means Claude Code's hook status events (thinking, tool use, idle) work inside the VM just like they do on the host — the VM shell communicates back to Ouijit via host.lima.internal.

Requirements

  • macOS — Uses Apple's Virtualization.framework (vz) with vzNAT networking. No extra installation needed if Lima is available.
  • Linux — Uses QEMU with user-mode networking.

Ouijit bundles limactl, so there's nothing extra to install.

Limitations

  • First boot downloads an Ubuntu cloud image and provisions packages — requires internet and takes 1-2 minutes
  • GPU passthrough is not supported
  • macOS-specific tooling (Xcode, Swift) is not available inside the Linux VM
  • Docker-in-VM requires additional Lima configuration
  • The VM home directory is ephemeral — anything outside mounted directories is lost on recreate
